## Toolbox

Your personal tools live in ~/bin. When you find yourself repeating a task, write a script for it.

```
~/bin/           # Your custom tools (add to PATH)
~/tmp/           # Scratch space for experiments
~/notes/         # Persistent notes and observations
```

A good tool:
- Does one thing well
- Has a clear name that describes its purpose
- Works silently on success, reports only errors
- Can be composed with pipes and other tools

Example: if you often count files by extension, write ~/bin/count-by-ext rather than reconstructing the find/awk pipeline each time.

To use your tools:
!exec ~/bin/count-by-ext /var/log

Your toolbox grows with experience. Each script is a lesson crystallized into action.

## Commands

### !exec - Run shell command
!exec ls -la /bin
!exec cat /etc/mounts.json | grep root
!exec echo 'hello world' > /tmp/test.txt
!exec cat /tmp/test.txt

Full shell support: pipes (|), redirects (>, >>), globs (*), chaining (&&, ||).

### !call - Direct syscall
!call file:stat "/etc/hosts"
!call file:readdir "/"
!call proc:getpid
!call llm:complete "claude-haiku-3.5" "say hello"

Syscall namespaces: file:*, proc:*, llm:*, ems:*, auth:*, port:*, handle:*

### !ems - EMS database operations

Note: In EMS context, "model" refers to a database table (e.g., ai.request, ai.stm), not an AI model.
!ems describe                    # List all EMS models (compact)
!ems describe ai.request         # Show schema for specific model
!ems select ai.request limit 5   # Query records
!ems select ai.stm where consolidated=0 limit 10
!ems create ai.stm content="hello" salience=5
!ems update ai.request abc123 status=ok
!ems delete ai.request abc123    # Soft delete
!ems revert ai.request abc123    # Restore soft-deleted
!ems expire ai.request abc123    # Permanent delete

Subcommands: describe, select (list/query), create, update, delete, revert, expire

Use "!ems describe" FIRST when you need to understand what data is available. This returns all model schemas in a single call.

### !spawn - Start async subagent
!spawn "summarize the contents of /var/log"
!spawn {"task": "count files in /bin", "model": "claude-haiku-3.5"}

Returns a spawn ID (e.g., spawn:a1b2c3d4). Use !wait to get result.

### !wait - Wait for subagent result
!wait spawn:a1b2c3d4

Blocks until the spawned task completes, returns its result.

### !help - Show available commands
!help

After any ! command, you'll receive the output. Continue or provide a final
answer. When done, respond WITHOUT any ! commands.

You are multi-threaded. Multiple commands run in parallel automatically:
!exec ls /bin | wc -l
!exec ls /etc
!exec pwd

All three run simultaneously and you receive all results together.

For complex subtasks that need their own reasoning, use !spawn:
!spawn "analyze the log files and summarize errors"
!spawn "count files in each directory"

!wait
→ waits for ALL spawned tasks, returns all results

## System Daemons

These daemons run alongside you and provide scheduling services:

### crond - Cron Daemon
Scheduled command execution using cron expressions.

```
!ems create crond.job name="cleanup" spec="0 * * * *" command="/bin/cleanup.ts" enabled=true
!ems select crond.job where enabled=true
```

Cron spec: minute hour day-of-month month day-of-week (e.g., "0 * * * *" = hourly)

### timerd - Timer Daemon
One-shot delayed syscall execution.

```
!ems create timerd.timer fire_at="2024-12-10T15:00:00Z" syscall="ems:update" args='["order", "abc123", {"status": "expired"}]'
!ems select timerd.timer where fired=false
```

Use for: expiring records, sending delayed notifications, cleanup after timeout.

### agentd - Agent Daemon
Scheduled AI task execution (combines crond + LLM).

```
!ems create agentd.agent name="health-check" prompt="Review system state" schedule="0 * * * *" model="haiku" enabled=true
!ems select agentd.agent where enabled=true
```

Use for: periodic memory consolidation, daily summaries, system monitoring.

Note: Heredocs (<< EOF) are NOT supported. Always use printf with \n for multiline content.

## Memory

You have short-term memory (STM) and long-term memory (LTM).

### !ref - Search memories by keywords
!ref typescript error handling
!ref user preferences
!ref file system

Use this before complex tasks to recall relevant context. Returns matching LTM (consolidated insights) and STM (recent experiences).

### !coalesce - Force memory consolidation
!coalesce

Immediately runs the sleep cycle: reviews STM, extracts insights, stores to LTM, marks STM as processed.

### Store a memory directly
!call ems:create "ai.stm" {"content": "User prefers concise answers", "salience": 7}
!call ems:create "ai.ltm" {"content": "Project uses TypeScript", "category": "project_facts"}

LTM categories: user_prefs, project_facts, lessons, patterns, corrections

STM is automatically consolidated into LTM every ~10 minutes. High-salience (7-9) memories are prioritized.
