# Phase 6.1: Core Entities Table

## Problem

Currently, each model table (file, folder, temp, etc.) duplicates hierarchy columns:
- `id`, `parent`, `name` appear in every table
- EntityCache must scan ALL model tables to build path index
- No single source of truth for entity identity

## Solution

Add a core `entities` table that owns identity and hierarchy:

```
entities (system table)
├── id          PRIMARY KEY
├── model       'file', 'folder', 'temp', etc.
├── parent      FK → entities.id
├── name        entity name
├── created_at
├── updated_at
└── trashed_at

file (detail table)
├── id          FK → entities.id (PRIMARY KEY)
├── owner
├── size
├── mimetype
└── checksum

folder (detail table)
├── id          FK → entities.id (PRIMARY KEY)
└── owner
```

## Benefits

1. **Single-table cache load** - `SELECT id, model, parent, name FROM entities`
2. **Guaranteed schema** - Every entity has id, model, parent, name
3. **Foreign key integrity** - `parent` references `entities.id`
4. **Model tables are pure detail** - Only model-specific fields
5. **Simpler path resolution** - No cross-table joins for hierarchy

## Schema Changes

### New: entities table

```sql
-- =============================================================================
-- ENTITIES TABLE (Core Identity + Hierarchy)
-- =============================================================================
-- Single source of truth for entity identity and path hierarchy.
-- This table is NOT a model - it's core infrastructure like models/fields.
--
-- WHY separate from model tables:
-- 1. EntityCache loads from ONE table, not all model tables
-- 2. Parent FK can reference any entity type (file in folder, link in folder)
-- 3. Path resolution is pure entities traversal, no model knowledge needed
--
-- WHY not seeded into models/fields:
-- This is infrastructure, not a user-visible entity type. Like models/fields
-- themselves, it exists outside the normal model system.

CREATE TABLE IF NOT EXISTS entities (
    -- 4 columns only - minimal for cache efficiency
    id          TEXT PRIMARY KEY DEFAULT (lower(hex(randomblob(16)))),
    model       TEXT NOT NULL,
    parent      TEXT REFERENCES entities(id),
    name        TEXT NOT NULL
);

-- Index for path resolution: find child by parent + name
CREATE UNIQUE INDEX IF NOT EXISTS idx_entities_parent_name
    ON entities(parent, name);

-- Index for listing children of a parent (readdir)
CREATE INDEX IF NOT EXISTS idx_entities_parent
    ON entities(parent);

-- Index for finding all entities of a model type
CREATE INDEX IF NOT EXISTS idx_entities_model
    ON entities(model);
```

### Modified: Model detail tables

Detail tables now have timestamps (entities table does not):

```sql
-- =============================================================================
-- FILE TABLE (Detail - has timestamps + model-specific fields)
-- =============================================================================
CREATE TABLE IF NOT EXISTS file (
    id          TEXT PRIMARY KEY REFERENCES entities(id) ON DELETE CASCADE,
    created_at  TEXT DEFAULT (datetime('now')),
    updated_at  TEXT DEFAULT (datetime('now')),
    trashed_at  TEXT,
    expired_at  TEXT,

    owner       TEXT NOT NULL,
    size        INTEGER DEFAULT 0,
    mimetype    TEXT,
    checksum    TEXT
);

-- =============================================================================
-- FOLDER TABLE (Detail)
-- =============================================================================
CREATE TABLE IF NOT EXISTS folder (
    id          TEXT PRIMARY KEY REFERENCES entities(id) ON DELETE CASCADE,
    created_at  TEXT DEFAULT (datetime('now')),
    updated_at  TEXT DEFAULT (datetime('now')),
    trashed_at  TEXT,
    expired_at  TEXT,

    owner       TEXT NOT NULL
);

-- =============================================================================
-- TEMP TABLE (Detail)
-- =============================================================================
CREATE TABLE IF NOT EXISTS temp (
    id          TEXT PRIMARY KEY REFERENCES entities(id) ON DELETE CASCADE,
    created_at  TEXT DEFAULT (datetime('now')),
    updated_at  TEXT DEFAULT (datetime('now')),
    trashed_at  TEXT,
    expired_at  TEXT,

    owner       TEXT NOT NULL,
    size        INTEGER DEFAULT 0,
    mimetype    TEXT
);

-- Similar for device, proc, link...
```

### Seed: Root entity

```sql
-- =============================================================================
-- SEED: ROOT ENTITY
-- =============================================================================
-- The root entity is the namespace origin. All paths start here.
-- WHY well-known UUID: Simplifies bootstrap, no discovery needed.

INSERT OR IGNORE INTO entities (id, model, parent, name) VALUES
    ('00000000-0000-0000-0000-000000000000', 'folder', NULL, '');

-- Root folder detail
INSERT OR IGNORE INTO folder (id, owner) VALUES
    ('00000000-0000-0000-0000-000000000000', 'system');
```

## EntityCache Changes

```typescript
// Before: Scan all model tables
async loadFromDatabase(db: DatabaseConnection): Promise<void> {
    const models = await db.query("SELECT model_name FROM models ...");
    for (const { model_name } of models) {
        const entities = await db.query(`SELECT id, parent, name FROM ${model_name} ...`);
        // ...
    }
}

// After: Single table query (no trashed_at filter!)
async loadFromDatabase(db: DatabaseConnection): Promise<void> {
    const entities = await db.query<CachedEntity>(
        "SELECT id, model, parent, name FROM entities"
    );
    for (const entity of entities) {
        this.addEntity(entity);
    }
}
```

**Key insight:** The cache contains ALL entities, including trashed ones. Soft-delete status is determined by querying the detail table's `trashed_at` field. This means:
- Trashing an entity doesn't invalidate the cache
- Path resolution works for trashed entities (detail query reveals status)
- Cache is stable and doesn't need to track lifecycle

## Observer Pipeline Changes

### Ring 5 (SQL Persistence)

Currently writes to model table only. New flow:

```typescript
// For CREATE:
// 1. Insert into entities (id, model, parent, name)
// 2. Insert into {model} detail table (id, ...model-specific fields)

// For UPDATE:
// 1. If parent/name changed: UPDATE entities
// 2. If model-specific fields changed: UPDATE {model}

// For DELETE:
// 1. UPDATE entities SET trashed_at = now()
// 2. Detail table row cascades or stays (for audit)
```

### Ring 8 (EntityCacheSync)

No change needed - already syncs on (id, model, parent, name).

## Migration Path

1. Create `entities` table
2. Populate from existing model tables:
   ```sql
   INSERT INTO entities (id, model, parent, name, created_at, updated_at, trashed_at)
   SELECT id, 'file', parent, name, created_at, updated_at, trashed_at FROM file;
   -- Repeat for folder, temp, device, proc, link
   ```
3. Drop hierarchy columns from model tables
4. Add FK constraints
