# Phase 6.1: Core Entities Table

## Problem

Currently, each model table (file, folder, etc.) duplicates hierarchy columns:
- `id`, `parent`, `pathname` appear in every table
- EntityCache must scan ALL model tables to build path index
- No single source of truth for entity identity

## Solution

Add a core `entities` table that owns identity and hierarchy:

```
entities (system table - 4 columns only)
├── id          PRIMARY KEY
├── model       'file', 'folder', 'user', etc.
├── parent      TEXT (no FK - avoids boot ordering issues)
└── pathname    derived from models.pathname field

file (detail table)
├── id          PRIMARY KEY (same as entities.id, no FK)
├── created_at
├── updated_at
├── trashed_at
├── expired_at
├── owner
├── size
├── mimetype
└── checksum

folder (detail table)
├── id          PRIMARY KEY (same as entities.id, no FK)
├── created_at
├── updated_at
├── trashed_at
├── expired_at
└── owner
```

## Benefits

1. **Single-table cache load** - `SELECT id, model, parent, pathname FROM entities`
2. **Guaranteed schema** - Every entity has id, model, parent, pathname
3. **Model tables are pure detail** - Only timestamps + model-specific fields
4. **Simpler path resolution** - No cross-table joins for hierarchy
5. **Polymorphic hierarchy** - Any model can be a child of any other model
6. **No boot ordering** - No FKs means tables initialize independently

## Schema Changes

### New: entities table

```sql
-- =============================================================================
-- ENTITIES TABLE (Core Identity + Hierarchy)
-- =============================================================================
-- Single source of truth for entity identity and path hierarchy.
-- This table is NOT a model - it's core infrastructure like models/fields.
--
-- WHY 4 columns only:
-- 1. Minimal for cache efficiency (~200 bytes per entity)
-- 2. Timestamps live in detail tables (soft-delete doesn't invalidate cache)
-- 3. Cache contains ALL entities, trashed or not
--
-- WHY separate from model tables:
-- 1. EntityCache loads from ONE table, not all model tables
-- 2. Parent FK can reference any entity type (polymorphic hierarchy)
-- 3. Path resolution is pure entities traversal, no model knowledge needed
--
-- WHY not seeded into models/fields:
-- This is infrastructure, not a user-visible entity type. Like models/fields
-- themselves, it exists outside the normal model system.

CREATE TABLE IF NOT EXISTS entities (
    id          TEXT PRIMARY KEY DEFAULT (lower(hex(randomblob(16)))),
    model       TEXT NOT NULL,
    parent      TEXT,  -- no FK, avoids boot ordering issues
    pathname    TEXT NOT NULL  -- derived from models.pathname field
);

-- Index for path resolution: find child by parent + pathname
CREATE UNIQUE INDEX IF NOT EXISTS idx_entities_parent_pathname
    ON entities(parent, pathname);

-- Index for listing children of a parent (readdir)
CREATE INDEX IF NOT EXISTS idx_entities_parent
    ON entities(parent);

-- Index for finding all entities of a model type
CREATE INDEX IF NOT EXISTS idx_entities_model
    ON entities(model);
```

### Modified: Model detail tables

Detail tables have timestamps + model-specific fields (entities table does not):

```sql
-- =============================================================================
-- FILE TABLE (Detail - has timestamps + model-specific fields)
-- =============================================================================
CREATE TABLE IF NOT EXISTS file (
    id          TEXT PRIMARY KEY,  -- same as entities.id, no FK
    created_at  TEXT DEFAULT (datetime('now')),
    updated_at  TEXT DEFAULT (datetime('now')),
    trashed_at  TEXT,
    expired_at  TEXT,

    owner       TEXT NOT NULL,
    size        INTEGER DEFAULT 0,
    mimetype    TEXT,
    checksum    TEXT
);

-- =============================================================================
-- FOLDER TABLE (Detail)
-- =============================================================================
CREATE TABLE IF NOT EXISTS folder (
    id          TEXT PRIMARY KEY,  -- same as entities.id, no FK
    created_at  TEXT DEFAULT (datetime('now')),
    updated_at  TEXT DEFAULT (datetime('now')),
    trashed_at  TEXT,
    expired_at  TEXT,

    owner       TEXT NOT NULL
);

-- Similar pattern for all VFS-addressable models...
```

### Seed: Root entity

```sql
-- =============================================================================
-- SEED: ROOT ENTITY
-- =============================================================================
-- The root entity is the namespace origin. All paths start here.
-- WHY well-known UUID: Simplifies bootstrap, no discovery needed.

INSERT OR IGNORE INTO entities (id, model, parent, pathname) VALUES
    ('00000000-0000-0000-0000-000000000000', 'folder', NULL, '');

-- Root folder detail
INSERT OR IGNORE INTO folder (id, owner) VALUES
    ('00000000-0000-0000-0000-000000000000', 'system');
```

## EntityCache Changes

```typescript
// Before: Scan all model tables
async loadFromDatabase(db: DatabaseConnection): Promise<void> {
    const models = await db.query("SELECT model_name FROM models ...");
    for (const { model_name } of models) {
        const entities = await db.query(`SELECT id, parent, pathname FROM ${model_name} ...`);
        // ...
    }
}

// After: Single table query (no trashed_at filter!)
async loadFromDatabase(db: DatabaseConnection): Promise<void> {
    const entities = await db.query<CachedEntity>(
        "SELECT id, model, parent, pathname FROM entities"
    );
    for (const entity of entities) {
        this.addEntity(entity);
    }
}
```

**Key insight:** The cache contains ALL entities, including trashed ones. Soft-delete status is determined by querying the detail table's `trashed_at` field. This means:
- Trashing an entity doesn't invalidate the cache
- Path resolution works for trashed entities (detail query reveals status)
- Cache is stable and doesn't need to track lifecycle

## Observer Pipeline Changes

### Ring 5 (SQL Persistence)

Currently writes to model table only. New flow:

```typescript
// For CREATE:
// 1. Insert into entities (id, model, parent, pathname)
// 2. Insert into {model} detail table (id, timestamps, ...model-specific fields)

// For UPDATE:
// 1. If parent changed: UPDATE entities SET parent = ?
// 2. If pathname source field changed: UPDATE entities SET pathname = ?
// 3. Other fields: UPDATE {model} detail table

// For DELETE (soft):
// 1. UPDATE {model} SET trashed_at = now()
// 2. entities table unchanged (cache stays valid)

// For DELETE (hard):
// 1. DELETE FROM entities WHERE id = ?
// 2. DELETE FROM {model} WHERE id = ? (no cascade, explicit delete)
```

### Ring 8 (EntityCacheSync)

Syncs cache on (id, model, parent, pathname) changes:
- CREATE: Add entity to cache
- UPDATE: If parent/pathname changed, update cache indexes
- DELETE (hard): Remove from cache

## Migration Path

1. Create `entities` table
2. Populate from existing model tables:
   ```sql
   INSERT INTO entities (id, model, parent, pathname)
   SELECT id, 'file', parent, filename FROM file;

   INSERT INTO entities (id, model, parent, pathname)
   SELECT id, 'folder', parent, name FROM folder;
   -- Repeat for other VFS-addressable models
   ```
3. Drop hierarchy columns from model tables (keep timestamps + model-specific fields)
